Motivation
The Privacy Problem
Why federated learning alone is not enough — and exactly what leaks from gradient updates.
The setting — keyboard next-word prediction
SECTION 1–2

Your phone keyboard predicts the next word you type. To improve this model Google needs data from millions of users — but the data is private text messages.

Federated learning keeps raw data on the device. Each phone trains locally and sends only a gradient update. Sounds private — but the paper shows the gradient update itself leaks private data.

The exact attack — how words are leaked
SECTION 2
Words are represented as one-hot vectors:
Vocabulary: [hello] [pizza] [password] [bank] [love] If you typed "password": x = [ 0, 0, 1, 0, 0 ] ↑ 1 at position of typed word, 0 everywhere else
The gradient for each word is non-zero ONLY when that word appeared in your data:
Gradient = [ 0.000, 0.000, 0.047, 0.000, 0.000 ] ↑ server sees this is non-zero → server knows you typed "password" → your private messages are exposed
The core privacy violation
The server only needs the sum of all gradients to update the model. It does not need individual gradients at all. But currently it receives each person's gradient separately — learning private information it was never supposed to see.

Why it is hard to fix — four mobile constraints
SECTION 2
Constraint What it means
High-dimensional vectors Neural networks have millions of parameters — updates are huge
Expensive communication Mobile data costs money — overhead must stay below 2×
Frequent dropouts Mobile users disconnect constantly — protocol must survive
Untrusted server All communication goes through server — server may be malicious
No existing solution addressed all four together. That gap motivated this paper.

Mechanism
Step 1 — Basic Pairwise Masking
The core trick — masks that cancel perfectly in the sum. Click through each step.
The idea — one adds, the other subtracts
SECTION 4.0.1
Each pair of users shares a random number. One user adds it to their value, the other subtracts it. When the server adds everyone's values together, the shared number appears once positive and once negative — they cancel out perfectly. Only the real values survive in the sum.
Shared numbers between pairs:
Alice and Bob share → 100
Alice and Carol share → 30
Bob and Carol share → 50
Rule:
smaller user number ADDS, larger user number SUBTRACTS Alice = user 1 (smallest) → always adds Bob = user 2 (middle) → adds to Carol, subtracts from Alice Carol = user 3 (largest) → always subtracts
Step by step with numbers
INTERACTIVE
Alice — adds both her shared numbers:
Alice's value = 5 + 100 + 30 = 135
Bob — subtracts Alice's shared number, adds Carol's:
Bob's value = 3 − 100 + 50 = −47
Carol — subtracts both shared numbers:
Carol's value = 7 − 30 − 50 = −73
Server receives and sums
135
+ (−47)
+ (−73)
= 15 ✓

Why? Shared numbers cancelled:
100 → Alice added +100, Bob subtracted −100 → zero ✓
30 → Alice added +30, Carol subtracted −30 → zero ✓
50 → Bob added +50, Carol subtracted −50 → zero ✓

Server saw 135, −47, −73. Never saw 5, 3, or 7.
Server got the correct answer
15
Without ever seeing 5, 3, or 7 individually
Two problems with this basic approach
Problem 1: Exchanging full mask vectors is too expensive for mobile — needs to be fixed.
Problem 2: If any user drops out mid-protocol, their masks are never cancelled and the sum is wrong.
Mechanism
Step 2 — Diffie-Hellman + PRG
How users agree on shared mask numbers cheaply — and how one short number becomes a full mask vector.
Diffie-Hellman — agreeing on a shared secret
SECTION 3.2
Alice and Bob need to agree on a shared number without the server finding out. Diffie-Hellman lets two people agree on a secret number by exchanging public values — even if the server sees all the messages, it cannot figure out the shared secret.
Simple numbers example:
Public number everyone knows:
g = 3
Alice picks a secret: a = 2 → sends 3² = 9
to Bob
Bob picks a secret: b = 4 → sends 3⁴ = 81
to Alice
Server sees: 9 and 81 — cannot figure out the shared secret
Alice computes: 81² = 6561
Bob computes: 9⁴ = 6561
Both get the same number 6561 — without ever sending it! This 6561 becomes their shared mask seed.
PRG — turning one number into millions
SECTION 3.4
The gradient vector has millions of numbers — so the mask also needs millions of numbers. But Diffie-Hellman only gives one shared number. A PRG (Pseudorandom Generator) solves this by expanding one short seed into as many numbers as needed.
Without PRG — too expensive:
Exchange full mask vector → millions of numbers per pair 1000 users → billions of numbers exchanged → impossible on mobile
With PRG — cheap:
Exchange one short seed (6561) per pair Alice: PRG(6561) → [100, 243, 891, 12, 557, ...] (millions of numbers) Bob: PRG(6561) → [100, 243, 891, 12, 557, ...] (same millions) Both get the SAME full mask vector from one short seed No extra communication needed — each expands locally
Communication saving
Without PRG: ~1 TB of data for 1000 users.
With DH + PRG: ~256 KB for 1000 users.
This is why the paper achieves only 1.73× overhead over raw data.
Mechanism
Step 3 — Shamir Secret Sharing
How the protocol handles users dropping out — by splitting secrets into pieces before the protocol starts.
The dropout problem
PROBLEM
If Bob drops out after the masking phase, his shared masks with Alice (100) and Carol (50) are stuck in the sum — never cancelled. The server cannot get the correct answer. We need a way to recover Bob's mask seed even after he is gone.
Shamir secret sharing — the fix
SECTION 3.1
Before the protocol starts, Bob splits his DH private key into pieces using Shamir secret sharing. Each other user holds one piece. Any 2 pieces out of 3 can reconstruct the key. One piece alone reveals nothing.
Bob's DH private key = 300, random number = 20
Bob creates pieces
Alice gets: 300 + (20 × 1) = 320
Carol gets: 300 + (20 × 3) = 360
If Bob drops out:
Alice gives piece: 320
Carol gives piece: 360
Find random number: (360 − 320) ÷ (3 − 1) = 20
Find secret: 320 − (20 × 1) = 300 ✓
Server recomputes Bob's mask seeds → cancels them from sum → correct answer ✓
If only one piece available → infinite possible answers → reveals nothing
New problem this creates
NEW ATTACK
Server can cheat
Server lies and says Bob dropped out. Collects Bob's DH key pieces. Reconstructs Bob's mask seeds (100 and 50). Then Bob's delayed message −47 arrives. Server subtracts the masks and learns Bob's real value 3 directly. A malicious server can trigger this on purpose.
This is exactly why the paper introduces a second mask — the personal mask. That is the next step.
Key Innovation
Step 4 — Double Masking
The paper's key contribution — a second personal mask that stops the server from ever stripping a user's real value bare.
The fix — add a personal mask
SECTION 4.0.3
Each user adds a second private random number — their personal mask — on top of the pairwise masks. This personal mask is also Shamir-shared among all other users. Now the server must make a choice for each user — it can only remove one type of mask, never both.
New masked value formula:
value sent = real value + personal mask + pairwise masks
Alice: 5 + 7 + 100 + 30 = 142
Bob: 3 + 12 − 100 + 50 = −35
Carol: 7 + 4 − 30 − 50 = −69
Personal masks: Alice= 7, Bob= 12, Carol= 4
The exclusive choice — why server cannot cheat
SECURITY
For each user, surviving users give ONE type of piece — never both:
If a user DROPPED OUT: → give piece of their DH key → server removes their pairwise masks → personal mask still hidden
If user IS ALIVE: → give piece of their personal mask → server removes their personal mask → pairwise masks already cancelled
Result:
Real value always hidden behind at least one layer Server can never remove both layers for the same user
Verification — all three alive
Server gets: 142 + (−35) + (−69) = 38
Pairwise masks cancelled in sum ✓
Server reconstructs personal masks: 7 + 12 + 4 = 23
Removes them: 38 − 23 = 15 ✓

Trade-offs
What Does Privacy Cost?
Every privacy method has costs. Here is every trade-off the paper explicitly measures and discusses.
Accuracy vs privacy
ZERO LOSS
No accuracy loss from secure aggregation
The server computes the exact same sum as without privacy. No noise, no approximation. Model quality is identical to training without any privacy mechanism.
Combined with Differential Privacy — bonus improvement
If noise is added for extra privacy, using SecAgg in batches of n users reduces the noise variance by a factor of n. For 1024-user batches this gives 32× better accuracy compared to adding noise individually — for the same privacy guarantee.
Communication cost vs privacy
SECTION 7.1
Raw data — no privacy 1.0×
SecAgg — 1024 users, 1M-dim vectors 1.73×
SecAgg — 16K users, 16M-dim vectors 1.98×
As the gradient vector gets larger, the key exchange overhead becomes a smaller fraction of the total. Overhead approaches a constant ~2× — within the paper's stated target.
Computation cost vs privacy
SECTION 7.2
m = length of data vector, n = total users, d = dropped users
Who Computation Communication Storage
Each user O(n² + mn) O(n + m) O(n + m)
Server O(mn²) O(n² + mn) O(n² + m)
Server cost with dropouts — Figure 8
500 users, 0% dropout → server takes ~2 seconds.
500 users, 30% dropout → server takes ~143 seconds.
Each dropped user forces many extra computations on the server side.
The Asymmetric Server Cost Equations
Cost = O(m(n - d) + md(n - d))
The server's computational penalty is highly asymmetric. Processing a dropped user forces the server to compute (n-d) PRG expansions to rebuild and manage the pairwise masks. So, dropouts are actively punished by the design of the system
Model complexity vs constraints
NO CONSTRAINT
Works on any model, any architecture, any gradient vector size. No changes to hyperparameters or training. Only constraint is needing at least t users to survive each round.
Section 9 — Related Work
Why Existing Approaches Failed
Five families of prior work — and why each one could not meet all four mobile constraints together.
Five prior approaches and their problems
SECTION 9
1 — Multiple non-colluding servers
How: Spread trust across multiple independent servers — each sees only a piece.
Problem: Requires multiple truly independent servers. A single cloud provider like Google cannot use this. Much harder to deploy in practice.
2 — Generic Secure MPC
How: General protocols that can compute any function securely.
Problem: Each user must send a copy of their entire gradient vector to many other users. Communication cost is enormous — completely impractical for mobile devices.
3 — DC-nets (Dining Cryptographers Networks)
How: Pairwise XOR blinding — strong anonymity.
Fatal problem: If even one user drops out before sending their message, the entire protocol must restart from scratch. Zero dropout tolerance — useless on mobile.
4 — Pairwise additive masking (Ács & Castelluccia 2011) — closest prior work
How: Same basic pairwise masking idea — DH key exchange, additive masks.
Problem: Recovery phase is brittle. If extra users drop during recovery the protocol cannot continue. Repeating recovery risks leaking personal masks of surviving users.
This paper's fix: Shamir sharing + double masking handles dropouts at any stage safely.
5 — Threshold Homomorphic Encryption
How: Compute on encrypted values without decrypting.
Problem: Requires expensive trusted setup. Some variants require all users to respond — zero dropout tolerance.
Current State
Deployed? Superseded? Still Valid?
What happened after this paper was published in 2017.
Commercial deployment
IN PRODUCTION
✓ Deployed in Google Gboard — Android keyboard
The paper states they expect to deploy soon. This happened. The protocol runs in Google's Gboard keyboard on Android — one of the very few MPC protocols ever deployed at mobile scale. A follow-up paper in 2019 describes the production deployment challenges.
Follow-up research
EXTENDED NOT REPLACED
SecAgg+ (Bell et al., 2020)
The paper identified O(n²) pairwise communication as a known limitation. SecAgg+ fixes this using a random graph instead of all-pairs — reducing communication from O(n²) to O(n log n). The original security proofs remain valid.
Has it been proved wrong?
No. Security proofs are still considered valid. No subsequent work has broken the protocol or found errors. The paper has not been superseded — only extended.
Open problems the paper leaves unsolved
SECTION 8
Problem Status
O(n²) pairwise communication ✓ Solved by SecAgg+ in 2020
